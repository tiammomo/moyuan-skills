# 从需求到 Skill 设计表

这篇文档提供一份可以直接复制填写的设计表。

它的目标不是解释概念，而是帮你把一个模糊需求，快速翻译成一个符合本仓库“渐进式 skill + harness 预留”思路的结构。

## 怎么用这张表

推荐用法：

1. 把下面整段复制到需求文档、PR 描述或笔记里
2. 先填“需求拆解”“触发与边界”“路由设计”
3. 再填“渐进式分层”“校验闭环”“Harness 预留”
4. 最后再开始写实际文件

## 设计表模板

| --- | --- |

| 核心输入 | `<目录/文档/API/schema/...>` |
| 核心输出 | `<文件/报告/修改结果/计划结果>` |

### 1. 需求拆解

| 用户原话 | 任务类型 | 最小输入 | 验收输出 | 风险点 |
| --- | --- | --- | --- | --- |
| `<把 X 同步到 Y>` | `<sync-dir>` | `<源 + 目标>` | `<dry-run 或执行结果>` | `<覆盖/删除/权限>` |

### 2. 触发与边界

| 项目 | 填写 |
| --- | --- |
| `Use when ...` 触发语句 | `<5-10 个触发场景>` |

### 3. 路由设计

| 场景 | 主入口动作 | 读取哪个 reference | 运行哪个 script | 风险控制 |
| --- | --- | --- | --- | --- |
| `<同步单文件>` | `<plan/push/pull>` | `references/<main>.md` | `scripts/<tool>.py` | `<dry-run 默认开启>` |

### 4. 渐进式分层

| 内容类型 | 放哪里 | 原因 |
| --- | --- | --- |
| 触发短语 | frontmatter | `<必须始终可见>` |
| 安全边界 | `SKILL.md` | `<触发后第一时间可见>` |
| 长说明 | `references/` | `<只在相关任务读取>` |
| 稳定执行逻辑 | `scripts/` | `<重复复用且需要稳定性>` |
| 模板/样例 | `assets/` | `<用于输出而不是上下文>` |

### 5. 校验闭环

| 检查项 | 命令 | 通过标准 |
| --- | --- | --- |
| 结构 lint | `python scripts/check_progressive_skills.py` | skill 结构与 reference 路由都通过 |
| skill 本地 smoke | `python skills/<skill>/scripts/check_<skill>.py` | 关键资源存在且格式正确 |
| 代表性任务验证 | `<你的命令>` | 至少一条主链路可跑通 |

## 提交前快速检查

- 触发是否足够窄？
- `SKILL.md` 是否仍然短小？
- reference 是否都是按需入口？
- script 是否真的值得存在？
- 是否留下了 skill-local 检查？
- 是否已经思考未来要不要进入 harness？
