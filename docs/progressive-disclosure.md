# 渐进式披露设计

这篇文档讨论的是：当一个 skill 越来越复杂时，信息应该怎样分层，才能既不丢失能力，也不把上下文拖垮。

## 核心原则

渐进式披露不是“把文档拆散”。

它真正要解决的是三件事：

1. 触发时只暴露最关键的信息
2. 执行时只读取当前任务真正需要的知识
3. 演进时能清楚知道一段内容应该放在哪一层

## 六层分工表

| 层 | 作用 | 适合放什么 | 不适合放什么 |
| --- | --- | --- | --- |
| frontmatter | 触发 | 技能目标、触发场景 | 长解释、示例、接口细节 |
| `SKILL.md` | 路由 | 安全边界、路径选择、默认工作流 | 大段背景、专题细节 |
| `references/` | 按需知识 | 长规则、接口、专题说明 | 核心 trigger、必读安全边界 |
| `scripts/` | 稳定执行 | CLI、解析器、批处理、验证脚本 | 只讲原理不执行的说明 |
| `assets/` | 输出资源 | 模板、样例、骨架文件 | 需要加载进上下文的知识 |
| harness | 系统外层 | eval、state、memory、tool contract、automation | 只属于单个 skill 的窄知识点 |

## 一个简单判断法

当你不确定一段内容放哪里时，可以连续问自己三个问题：

1. 这段内容是不是“每次触发都必须看到”？
   如果是，优先考虑 frontmatter 或 `SKILL.md`
2. 这段内容是不是“只在某一类任务里才需要”？
   如果是，优先考虑 `references/`
3. 这段内容是不是“重复执行且需要稳定输出”？
   如果是，优先考虑 `scripts/`

## 什么时候应该从 `SKILL.md` 拆出去

出现这些信号时，通常该拆：

- `SKILL.md` 已经开始解释多个专题细节
- 同一份 `SKILL.md` 同时服务多条差异很大的路径
- agent 触发 skill 后，真正该看哪段反而不明显
- 修改一个细节时，不知道该改正文、脚本还是参考资料

## 好的切分，不只是“变短”

一个好的渐进式结构至少要满足：

- 触发变准了
- 路由变清楚了
- 修改成本变低了
- 校验入口变明确了

如果只是把长文档切成很多小文档，但 agent 仍然不知道先看哪个入口，那不叫渐进式披露，只是内容碎片化。

## Router 设计原则

好的 `Task Router` 应该做到：

- 用场景分，而不是用文件名分
- 每条路由都说明“什么时候看哪个 reference”
- 每条路由优先只有一个主入口
- 危险操作要在 router 里就暴露 guardrail

## 长 reference 的处理方式

当一个 reference 超过 100 行时，建议：

1. 在顶部补 `## Contents`
2. 覆盖后续所有二级标题
3. 把最常见入口放在前面

这不是为了好看，而是为了让 agent 在预览时能更快定位到最窄段落。

## 三个常见反模式

### 1. 什么都留在 `SKILL.md`

结果：

- 技能触发后加载太重
- 真正的入口不突出
- 每次改动都像改总说明书

### 2. 什么都拆到 `references/`

结果：

- 核心安全边界被藏深了
- 路由本身不清楚
- agent 需要额外猜“先看哪一份”

### 3. 把执行逻辑写成说明文字

结果：

- 相同代码反复手写
- 行为不稳定
- 很难回归

## 一个实用的重构顺序

当你面对一个臃肿 skill，推荐按这个顺序重构：

1. 先收缩 frontmatter
2. 再把 `SKILL.md` 改回最小入口
3. 再把长说明拆到 `references/`
4. 再把重复实现沉到 `scripts/`
5. 最后补 `assets/` 和检查脚本

## 渐进式披露和 harness 的关系

渐进式披露解决的是“知识如何被按需加载”。

harness 解决的是“系统如何围绕这些知识稳定运行”。

一个好的 skill 通常是一个好 harness 的前提，因为如果内容边界不清，后续就无法明确区分：

- 哪些是 prompt/router 问题
- 哪些是 tool/state/eval 问题

所以，渐进式披露不是文档技巧，而是系统设计前置动作。
