# Review Checklist

## Before Publishing

Check these items in order:

- product name, version, and date are correct
- summary matches the real source changes
- no confidential or unapproved detail leaked into public notes
- breaking changes are explicit and not buried
- known issues are accurate and still current
- placeholder text is gone

## Audience Check

For internal release notes:

- tickets are usually acceptable
- operational detail may be useful

For external release notes:

- remove irrelevant internal detail
- translate implementation language into user-facing value

## Final Safety Check

Always do at least one of these before publishing:

- run the bundled linter
- do a human editorial pass

The script helps with structure, but publication still needs judgment.
