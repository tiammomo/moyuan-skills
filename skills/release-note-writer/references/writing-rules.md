# Writing Rules

## Summary

The summary should answer:

- what changed
- who should care
- why this release matters

Keep it factual and compact.

## Highlights

Highlights should focus on the most user-visible or operationally important changes.

Avoid:

- internal implementation detail with no audience impact
- vague phrases like "various improvements"

Prefer:

- specific outcomes
- specific workflows that changed

## Grouped Sections

The section headings in the template are deliberate:

- Features
- Improvements
- Fixes
- Breaking Changes
- Known Issues

Keep bullet points consistent and easy to scan.

## Tickets and Impact

Tickets are helpful in internal notes and review drafts.

For public-facing notes, keep them only when the audience expects traceability.

Use impact text when it clarifies migration cost, changed behavior, or follow-up action.
